# AI Usage

## Which agent/tools were used

Claude (Anthropic), chat-based.
Осознанно не IDE-агент: код применяется руками, каждый кусок проходит ревью
до попадания в репозиторий.

Цикл работы: я формулирую шаг → AI выдаёт спецификацию или код → я применяю → делаю коммит 
→ AI ревьюит код из репозитория прогонами (клонирует, гоняет проверочные скрипты, pytest, tsc с моками зависимостей) → 
фиксим до зелёного → следующий шаг.

Один шаг = один коммит, история изменений отражает процесс.

## Which parts were delegated

**Делегировано AI:**
- спецификации каждого шага (структура файлов, сигнатуры, критерии приёмки);
- Chrome Extension целиком — типы API-контракта, popup-компонент, стили
  (React/TypeScript вне моей экспертизы);
- тесты API вместе с моками сессии БД (conftest с FakeSessionMaker);
- контент валидного ответа fake-провайдера;
- черновики README и этого документа.

**Оставлено за собой:**
- архитектура и границы слоёв (api / service / providers / repository);
- схема БД: что именно хранить в run;
- ключевые трейд-оффы: create_all вместо миграций, синхронная обработка,
  HTTP 200 для failed-ранов, CORS `*` для прототипа;
- реализация всего бэкенда по спецификациям;
- финальное решение по каждому коммиту.

## Examples of prompts / task briefs

Рабочий формат промпта: роль → контекст → задача → ограничения → критерии
приёмки. Роль задана один раз в начале сессии («ты мой техлид: даёшь
спецификации шагов, ревьюишь мой код из репозитория, проверяешь прогонами»)
и действует на все промпты ниже.

**1. Разбор ТЗ и план**

- *Контекст:* полный текст тестового задания; вводные — бэкенд знаю,
  React/расширения не знаю, срок один день.
- *Задача:* подсветить подводные камни в формулировках задания и дать план
  на день с таймбоксами.
- *Ограничения:* минимальный вариант без переусложнения; учесть, что
  AI-workflow тоже оценивается и его надо документировать по ходу.
- *Критерии:* явные риски по каждому пункту задания; план, закрывающий все
  обязательные требования.
- → Выявлено: fake-провайдер — основной путь ревью (запуск без ключей), пять
  сценариев structured output — ядро оценки, AI_USAGE нельзя писать задним
  числом.

**2. Спецификация fake-провайдера**

- *Контекст:* слои уже зафиксированы (api / service / providers / repository);
  Pydantic-модель BriefDecodeResult готова и отвергает malformed / missing /
  invalid severity.
- *Задача:* интерфейс провайдера и fake-реализация со сценариями ошибок.
- *Ограничения:* `async decode_brief(text) -> str` — сырой текст, не dict:
  парсинг и валидация только в сервисе, путь общий для fake и реального
  провайдера; сценарии переключаются ключевым словом в начале текста
  (`FAIL:malformed` / `missing` / `severity` / `provider`), чтобы error state
  демонстрировался и из тестов, и из расширения.
- *Критерии:* каждый сценарий валится на своём типе ValidationError;
  severity — только на enum (не enum+missing); `FAIL:provider` — исключение,
  не возврат; валидный ответ проходит собственную схему.

**3. Тесты API с моками БД**

- *Контекст:* эндпоинты и сервис готовы; сервис берёт сессии из
  `async_session_maker` (begin() на запись, вызов на чтение).
- *Задача:* test_api через TestClient + conftest с моком персистентности.
- *Ограничения:* мокать только `async_session_maker` — провайдер настоящий
  fake, цепочка provider → validate → map тестируется реальная; без
  pytest-asyncio; тесты должны бегать без Postgres.
- *Критерии:* happy-path, оба failure-path (provider / validation), 422 на
  пустом тексте, roundtrip POST→GET (значит мок begin() и __call__ разделяют
  общее хранилище), 404; в ответах нет raw_output/input_text.

**4. Popup расширения**

- *Контекст:* WXT react-ts скаффолд (WXT 0.20, React 19), файл
  `entrypoints/popup/App.tsx`; контракт бэкенда — схема RunRead приложена;
  failed-ран приходит с HTTP 200 и error_code в теле.
- *Задача:* компонент попапа — textarea, кнопка запуска, рендер результата
  (summary, списки, риски с бейджами severity), copy JSON.
- *Ограничения:* без сторонних библиотек — только React и CSS; UI-состояния —
  discriminated union одним полем, не набор булевых флагов.
- *Критерии:* четыре состояния (idle / loading / success / error); кнопка
  disabled на пустом тексте и во время запроса; failed-ран и транспортная
  ошибка оба дают error-состояние с сообщением; tsc strict без ошибок.

**5. Точечные разборы для верификации чужого кода**

Отдельный класс промптов — не «сделай», а «объясни, почему»: «почему тесты
синхронные при async-приложении», «как scalars() работает с select», «в чём
разница default и server_default у колонки». Формат вырожденный: контекст —
конкретный кусок кода из репозитория, задача — механика его работы; без
ограничений и критериев. Роль таких промптов в workflow — не принимать код
(в том числе сгенерированный), пока не могу объяснить, как он работает.

## What was accepted, changed or rejected

- **Принято:** модель BriefDecodeResult (все поля обязательные — missing
  fields ловится самой схемой, severity — enum'ом); conftest-моки; popup
  после typecheck и ручной проверки.
- **Изменено:** VALID_RESPONSE fake-провайдера — первый вариант содержал
  плейсхолдер из ТЗ (`"severity": "low|medium|high"`) и не проходил
  собственную схему; заменён на осмысленный разбор. Сценарий invalid severity
  переделан на «порчу валидного JSON одной заменой», чтобы падал ровно на
  enum, а не enum+missing. CSS disabled-кнопки — нечитаемый контраст.
- **Отклонено:** паттерны из прошлого проекта (очереди, outbox, Alembic) как
  избыточные для синхронного прототипа; pytest-asyncio — не нужен при
  синхронных тестах через TestClient; наращивание специфичности CSS против
  конфликта со скаффолдом (вместо этого удалены шаблонные стили).

## How generated code was verified

- **Прогонами, не чтением:** каждый коммит AI проверял исполнением — все
  сценарии провайдера через Pydantic-модель, все ветки сервиса на моках,
  все коды API через TestClient. Несколько багов найдено именно так.
- **Тесты:** pytest — 11 passed, без внешних зависимостей.
- **Фронтенд:** tsc strict (React 19 / TS 5.9) — 0 ошибок; `npm run dev`;
  ручная проверка всех четырёх состояний попапа в Chrome.
- **Smoke в реальном окружении:** docker compose up --build + curl по всем
  эндпоинтам (health, decode valid, decode FAIL:provider, get run) — вручную,
  после зелёных тестов. Принцип: зелёные тесты с моками ≠ работающая система.

## Mistakes / limitations found in agent output

1. **Несуществующая CLI-команда:** AI выдал `npm create wxt@latest` — пакета
   create-wxt в npm нет (проверено запросом к registry); реальная команда —
   `npx wxt@latest init`.
2. **return вместо raise** в сценарии FAIL:provider первой версии фейка —
   объект исключения возвращался как значение и не ловился бы except'ом
   в сервисе. Найдено прогоном сценариев.
3. **Плейсхолдер вместо данных:** валидный ответ фейка, собранный из примера
   в ТЗ, не проходил собственную Pydantic-схему (см. Changed).
4. **Нечитаемое disabled-состояние кнопки** в сгенерированном CSS — найдено
   только глазами в браузере; typecheck и сборка такие дефекты не видят.
5. **Конфликт с контекстом проекта:** сгенерированный App.css конфликтовал с
   глобальными стилями WXT-шаблона (тёмная тема, свои правила для button).
   Дефект не в сгенерированном файле, а на стыке со скаффолдом — код от AI
   надо проверять в окружении, а не изолированно.
